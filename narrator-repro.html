Narrator aria-activedescendant Repro

Demonstrates that Windows Narrator does not announce aria-activedescendant changes in a combobox when role="application" is present on an ancestor. Tested on Windows 11 with Narrator + Microsoft Edge.

Test 1 — Baseline (no role="application") PASS with NVDA/JAWS

Press ↓ / ↑ inside the textarea to move through options. Expected: Narrator announces each option as selection changes.

Why role="application" is needed

Without role="application", pressing ↓ in the textarea causes Narrator to enter Scan mode, which intercepts all arrow keys and prevents the application from receiving them. The developer is therefore forced to choose between:

  • Suppress scan mode with role="application" → aria-activedescendant stops being announced
  • Remove role="application" → Arrow keys stolen by scan mode, app unusable

Workaround in use: document.ariaNotify() (Edge 136+, origin trial). This works but is not yet broadly available.

Reproduction steps

  1. Open this page in Microsoft Edge on Windows 11 with Narrator running
  2. Tab to the Test 1 textarea and press ↓ / ↑ — note Narrator announces each option
  3. Tab to the Test 2 textarea and press ↓ / ↑ — Narrator is silent
Environment:
  OS:       Windows 11
  Browser:  Microsoft Edge (latest)
  Narrator: Windows built-in

Relevant ARIA:
  ancestor:  role="application"
  input:     role="combobox" + aria-activedescendant="opt-N"
  popup:     role="listbox"  + li[role="option"]

Expected:  Narrator announces option label on aria-activedescendant change
Actual:    Narrator is silent; no announcement made
